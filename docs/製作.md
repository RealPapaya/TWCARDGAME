# TWCARDGAME v2 製作與維護指南

這份文件描述目前 v2 重寫版的工作方式。根目錄是 v2 workspace。

## 目標

- 遠端 1v1 PvP 使用 Colyseus authoritative server，不讓 client 直接改遊戲狀態。
- 所有遊戲規則集中在 deterministic rules engine，方便測試、重播、避免 desync。
- 卡牌資料以 repo 內檔案為 source of truth，新增卡牌要能被 schema 與測試驗證。
- UI 可以簡單，但資料流必須乾淨：public state 走 Colyseus state sync，私密手牌走個別 client message。

## 目錄

- `apps/server`: Colyseus server、room adapter、連線與私密訊息。
- `apps/web`: Vite + vanilla TypeScript client prototype。
- `packages/cards`: 卡牌 catalog、schema、effect type 清單、驗證。
- `packages/rules`: 純規則引擎。不得依賴 DOM、Colyseus、Supabase、timer、`Math.random()`。
- `packages/shared`: 共用型別，包括 command、event、public state。
- `packages/db`: Supabase helper 與 migration。
- `packages/test-utils`: 測試 fixture 與 seed match helper。
- `docs`: 架構與製作文件。

## 核心規則

所有 gameplay mutation 都必須走：

```txt
GameRoom.handleCommand -> reduce(...) -> rules state/events -> Colyseus sync
```

不要在 `apps/web` 直接模擬或套用權威狀態。前端只能：

- 發送 command。
- 渲染 Colyseus public state。
- 渲染 server 個別傳來的 private hand state。
- 播放 event animation。

不要在 `packages/rules` 使用：

- DOM/browser API
- Colyseus API
- Supabase API
- `Date.now()`
- `Math.random()`

需要時間與隨機時，從 command envelope 的 `nowMs` 與 match private PRNG state 取得。

```bash

npm run build
npm run start -w @twcardgame/server
npm run dev -w @twcardgame/web
```

- Web: `http://localhost:5173`

## 新增或修改卡牌

1. 修改 `packages/cards/src/catalog.generated.ts` 或後續正式 catalog 來源。
2. 如果新增 effect type，同步更新：
   - `packages/cards/src/types.ts`
   - `packages/cards/src/validation.ts`
   - `packages/rules/src/effects.ts`
   - tests
3. 執行：

```bash
npm run validate:cards
npm test
npm run check
```

新增卡牌時不要只讓 UI 顯示成功；必須讓 rules engine 真正支援該效果。

## 修改遊戲規則

優先改 `packages/rules`，並新增或更新測試。

最低驗證：

- 同 seed + 同 command log 結果一致。
- public state 不包含手牌內容與 deck order。
- out-of-turn 或 invalid command 會被拒絕。
- 影響 card effect 時，catalog 裡所有同類型卡牌仍通過測試。

## 修改 Colyseus Server

`apps/server` 只做 adapter 工作：

- 分配 seat。
- 建立 match。
- 收 command。
- 呼叫 `reduce(...)`。
- sync public schema。
- 用 direct message 傳 private hand。
- 處理 reconnect / leave。

不要把卡牌規則或戰鬥邏輯寫在 room 裡。

Colyseus Schema 注意事項：

- 使用 `defineTypes(...)`，避免 decorator metadata 與 TS emit 差異。
- `tsconfig.base.json` 保持 `useDefineForClassFields: false`，否則 schema accessor 可能被 class fields 蓋掉。

## 修改 Web Client

`apps/web` 是 prototype，不是最終美術版。改 UI 時仍要遵守：

- client 不自行改權威遊戲狀態。
- command 必須有 `commandId`。
- target 用 `TargetRef`。
- 手牌只來自 server 的 `hand` message。
